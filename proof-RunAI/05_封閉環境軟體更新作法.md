# 封閉環境（Air-Gap）下的軟體更新作法

## 1. 概述

在政府機關、金融機構或高度安全管制場域，AI 運算叢集通常部署於與外網完全隔離的封閉網路（Air-Gap）環境。本文件說明在此環境下，如何安全、有效地執行 Kubernetes、NVIDIA GPU Operator、Run.ai 及相關套件的軟體更新作業。

---

```

        │
```

```
        │

        │

        │

        │
```

## 4. 各元件更新作法

### 4.1 容器映像檔（Container Images）

#### 準備端（外網工作站）

```bash
# 步驟 1：確認需要更新的映像檔清單
# ⚠️ Run.ai v2.25 的平台映像清單請以官方 air-gapped 套件（§4.2 方法一）
#    或 helm template 輸出為唯一來源，不可使用舊版映像名稱。
#
# 應用層映像範例（不含平台元件）：
# - nvcr.io/nvidia/cuda:12.6.0-base-ubuntu22.04       ← CUDA 基底映像
# - nvcr.io/nvidia/tritonserver:25.03-py3              ← 推論伺服器
# - registry.k8s.io/kube-apiserver:v1.33.0             ← K8s 控制平面映像

# 步驟 2：拉取映像檔
docker pull nvcr.io/nvidia/cuda:12.6.0-base-ubuntu22.04
docker pull nvcr.io/nvidia/tritonserver:25.03-py3

# 步驟 3：打包為 tar 檔
docker save nvcr.io/nvidia/cuda:12.6.0-base-ubuntu22.04 \
  -o cuda-12.6.0-base-ubuntu22.04.tar

docker save nvcr.io/nvidia/tritonserver:25.03-py3 \
  -o tritonserver-25.03-py3.tar

# 步驟 4：產生 checksum
sha256sum *.tar > images.sha256
```

#### 掃描與驗證（傳輸工作站）

```bash
# 使用 Trivy 掃描映像檔（離線模式）
trivy image --input cuda-12.4.0-base-ubuntu22.04.tar \
  --offline-scan \
  --severity HIGH,CRITICAL \
  --exit-code 1   # CRITICAL 漏洞時返回錯誤

# 若通過，驗證 checksum
sha256sum -c images.sha256
```

#### 匯入端（封閉環境）

```bash
# 步驟 1：載入映像檔
docker load -i cuda-12.4.0-base-ubuntu22.04.tar

# 步驟 2：重新標記為私有倉庫路徑
docker tag nvcr.io/nvidia/cuda:12.4.0-base-ubuntu22.04 \
  harbor.internal.local/nvidia/cuda:12.4.0-base-ubuntu22.04

# 步驟 3：推送至私有 Harbor
docker push harbor.internal.local/nvidia/cuda:12.4.0-base-ubuntu22.04
```

#### 批次處理腳本（封閉環境匯入）

```bash
#!/bin/bash
REGISTRY="harbor.internal.local"
IMAGE_DIR="/mnt/transfer/images"

for tar_file in ${IMAGE_DIR}/*.tar; do
  echo "Loading: ${tar_file}"
  
  # 載入並取得映像名稱
  image_name=$(docker load -i "${tar_file}" | awk '/Loaded image:/{print $3}')
  
  # 解析 original registry 並替換為內部 registry
  internal_name="${REGISTRY}/$(echo ${image_name} | sed 's|[^/]*/||')"
  
  docker tag "${image_name}" "${internal_name}"
  docker push "${internal_name}"
  
  echo "Pushed: ${internal_name}"
done
```

### 4.2 Helm Charts

#### 準備端（外網）

#### 方法一（建議）：使用官方 Air-Gapped 套件

v2.25 起 NVIDIA 提供完整的 `runai-airgapped-package-<VERSION>.tar.gz`，包含所有 chart 與映像，是封閉環境的標準安裝方式，**不需要 helm repo add**。

```bash
# ── 準備端（外網）──
# 從 NGC 下載官方 air-gapped 套件
# 請至 NGC（https://ngc.nvidia.com）搜尋 "runai-airgapped-package"
# 或依 NVIDIA / Run.ai 提供的官方指引取得套件下載連結
# 套件命名格式：runai-airgapped-package-<VERSION>.tar.gz

# 下載後驗證完整性（checksum 由 NGC 頁面提供）
sha256sum runai-airgapped-package-<VERSION>.tar.gz > airgapped.sha256
```

```bash
# ── 封閉環境：解壓並直接使用 chart ──
tar xzf runai-airgapped-package-<VERSION>.tar.gz

# 套件目錄結構（參考，實際檔名以解壓後 ./chart/ 內容為準）：
#   ./chart/control-plane-<VERSION>.tgz   ← Control Plane chart
#   ./chart/runai-cluster-<VERSION>.tgz   ← Cluster chart
#   ./images/                              ← 所有平台映像 tar 包
#   ./scripts/                             ← 安裝輔助腳本

# 載入映像至 Harbor（依 ./scripts/ 內輔助腳本執行，或手動批次 docker load + push）

# 以實際 chart 檔名執行升級（不鎖死版本字串，避免 patch 版號不符）
CP_CHART=$(ls ./chart/control-plane-*.tgz | head -1)
CLUSTER_CHART=$(ls ./chart/runai-cluster-*.tgz | head -1)

# 升級 Control Plane（release name: runai-backend）
helm upgrade runai-backend "${CP_CHART}" \
  -n runai-backend \
  -f values-override.yaml

# 升級 Cluster（release name: runai）
helm upgrade runai "${CLUSTER_CHART}" \
  -n runai \
  -f cluster-values-override.yaml
```

#### 方法二（可選）：ChartMuseum 內部鏡像化

適合需要版本集中管理或多叢集分發的場景。先從 NGC 拉取 chart，再推入內部 ChartMuseum。

```bash
# ── 準備端（外網）：從 NGC Helm repo 拉取 chart ──
# NGC repo（依官方文件）：https://helm.ngc.nvidia.com/nvidia/runai
# ⚠️ chart 命名請以 NVIDIA 官方安裝文件最終確認為準
# 將 <VERSION> 替換為實際目標版本號（例如 2.25.10）
VERSION="<VERSION>"

helm repo add runai https://helm.ngc.nvidia.com/nvidia/runai
helm repo update
helm pull runai/control-plane  --version "${VERSION}"   # Control Plane chart
helm pull runai/runai-cluster  --version "${VERSION}"   # Cluster chart

sha256sum *.tgz > charts.sha256
```

```bash
# ── 封閉環境：匯入 ChartMuseum ──
# 以實際 chart 檔名為準，避免 patch 版號不符
CP_CHART=$(ls control-plane-*.tgz | head -1)
CLUSTER_CHART=$(ls runai-cluster-*.tgz | head -1)

curl -X POST \
  -F "chart=@${CP_CHART}" \
  http://chartmuseum.internal.local/api/charts

curl -X POST \
  -F "chart=@${CLUSTER_CHART}" \
  http://chartmuseum.internal.local/api/charts

# 升級指令（version 從 chart metadata 讀取，不依賴檔名格式）
CHART_VERSION=$(helm show chart "${CP_CHART}" | grep '^version:' | awk '{print $2}')

helm repo add internal-charts http://chartmuseum.internal.local
helm repo update

helm upgrade runai-backend internal-charts/control-plane \
  --version "${CHART_VERSION}" \
  -n runai-backend \
  -f values-override.yaml

helm upgrade runai internal-charts/runai-cluster \
  --version "${CHART_VERSION}" \
  -n runai \
  -f cluster-values-override.yaml
```

> ⚠️ **過渡注意**：v2.25 之前仍使用 `runai.jfrog.io` 的既有部署，升級至 v2.25 時必須先切換至 NGC，再執行升級。升級完成後停止引用 JFrog 路徑。

### 4.3 Kubernetes 版本升級

#### kubeadm 離線升級

```bash
# 準備端（外網）：下載 kubeadm、kubelet、kubectl 套件
# Run.ai v2.25 支援 K8s 1.33–1.35，以 v1.33 為例
# Ubuntu/Debian
apt-get download kubeadm=1.33.0-1.1 kubelet=1.33.0-1.1 kubectl=1.33.0-1.1

# 額外下載相依套件
apt-get download --dependencies kubeadm=1.33.0-1.1

# 打包
tar czf k8s-1.33.0-debs.tar.gz *.deb
sha256sum k8s-1.33.0-debs.tar.gz

# 封閉環境安裝
dpkg -i /mnt/transfer/debs/kubeadm_1.33.0-1.1_amd64.deb
dpkg -i /mnt/transfer/debs/kubelet_1.33.0-1.1_amd64.deb
dpkg -i /mnt/transfer/debs/kubectl_1.33.0-1.1_amd64.deb
```

#### 控制平面升級映像檔預載

```bash
# 確認升級所需映像清單
kubeadm config images list --kubernetes-version v1.33.0

# 於外網下載並打包（同 4.1 容器映像作法）
# 封閉環境中預載
kubeadm config images pull \
  --image-repository harbor.internal.local/google_containers \
  --kubernetes-version v1.33.0
```

### 4.4 NVIDIA GPU Operator 離線更新

GPU Operator 由多個元件映像檔組成，需完整打包：

```bash
# Run.ai v2.25 支援 GPU Operator 25.10 ~ 26.3，以 v26.3 為示範
# ⚠️ 版本號以 helm template 實際輸出為唯一可信來源，不可使用舊版映像清單

# 步驟 1：於外網產生完整映像清單（每次升級均須重新執行）
helm repo add nvidia https://helm.ngc.nvidia.com/nvidia
helm repo update

helm template gpu-operator nvidia/gpu-operator \
  --version 26.3.0 \
  --set driver.version=570.124.06 \
  | grep -oP 'image:\s*\K\S+' | sort -u \
  > gpu-operator-26.3.0-images.txt

# 步驟 2：批次拉取並打包（依清單逐一處理）
while IFS= read -r image; do
  echo "Pulling: ${image}"
  docker pull "${image}"
  safe_name=$(echo "${image}" | tr '/: ' '__')
  docker save "${image}" -o "${safe_name}.tar"
done < gpu-operator-26.3.0-images.txt

sha256sum *.tar > gpu-operator-26.3.0-images.sha256

# 步驟 3（封閉環境）：重新標記並推入 Harbor，再執行升級
helm upgrade gpu-operator internal-charts/gpu-operator \
  --version 26.3.0 \
  -n gpu-operator \
  --set driver.version=570.124.06 \
  --set operator.repository=harbor.internal.local/nvidia \
  --set driver.repository=harbor.internal.local/nvidia \
  --set devicePlugin.repository=harbor.internal.local/nvidia \
  --set dcgmExporter.repository=harbor.internal.local/nvidia \
  --set gfd.repository=harbor.internal.local/nvidia
```

> ⚠️ **GPU Operator 版本備註**：Run.ai v2.25 官方文件中存在版本不一致（部分段落仍寫 25.3–25.10）。依 Cluster System Requirements 及 What's New 雙重佐證，採用 **25.10–26.3**；正式標案請向 NVIDIA Support 取得書面確認後再鎖定版本。

### 4.5 Run.ai 離線授權更新

```bash
# 離線授權文件由 NVIDIA 提供（.jwt 或 .lic 格式）
# 透過 kubectl secret 更新

kubectl create secret generic runai-license \
  -n runai-backend \
  --from-file=license.jwt=/mnt/transfer/runai-offline-license.jwt \
  --dry-run=client -o yaml | kubectl apply -f -

# 重啟 Control Plane 使授權生效
kubectl rollout restart deployment -n runai-backend
```

## 5. OS 套件更新（本地 APT/YUM Mirror）

### 5.1 建立本地 APT Mirror（Ubuntu）

```bash
# 使用 apt-mirror 工具同步（於外網工作站或 DMZ）
cat /etc/apt/mirror.list
# deb https://download.docker.com/linux/ubuntu jammy stable
# deb https://pkgs.k8s.io/core:/stable:/v1.33/deb/ /

apt-mirror

# 打包並傳入封閉環境
rsync -av /var/spool/apt-mirror/mirror/ /mnt/nas/apt-mirror/
```

### 5.2 封閉環境節點設定本地源

```bash
# 修改 /etc/apt/sources.list.d/internal.list
deb [trusted=yes] http://apt-mirror.internal.local/ubuntu jammy main restricted
deb [trusted=yes] http://apt-mirror.internal.local/kubernetes jammy main

# 更新
apt-get update
apt-get upgrade -y
```

## 6. 更新包管理規範

### 6.1 傳輸介質管理

| 項目 | 規範 |
|------|------|
| 傳輸介質 | 專用加密 USB 或 SFTP 安全通道（不使用個人設備） |
| 完整性驗證 | 傳入前後均需比對 SHA-256 checksum |
| 病毒掃描 | 所有檔案通過防毒掃描後方可傳入 |
| 記錄留存 | 每次傳輸填寫《軟體更新傳輸紀錄單》，留存 1 年 |

### 6.2 更新版本控管

```
封閉環境軟體版本追蹤（範例，以 Run.ai v2.25 Support Matrix 為基準）

元件               目前版本      可用版本     計畫更新日期
──────────────────────────────────────────────────────
Kubernetes         v1.32.x      v1.33.x      2026-06-15
Run.ai             v2.24.x      v2.25.x      2026-06-15
GPU Operator       25.10.x      26.3.x       2026-06-20
NVIDIA Driver      570.86.x     570.124.x    2026-06-20
Harbor             v2.11.x      v2.12.x      2026-07-01
```

> ⚠️ GPU Operator 版本請在升級前向 NVIDIA Support 確認官方 Support Matrix 最終版本。

### 6.3 回滾程序

```
發現更新異常
      │
      ▼
立即停止更新作業
      │
      ▼
評估影響範圍（僅 Run.ai？或 K8s 整體？）
      │
      ├── Helm 回滾
      │   helm rollback runai-backend <revision>
      │   helm rollback runai <revision>
      │
      ├── K8s 版本回滾
      │   使用 Velero 備份還原（若 API 不相容）
      │
      └── 映像檔回滾
          docker tag <old-image> harbor.internal.local/xxx:current
          kubectl rollout restart deployment -n <namespace>
      │
      ▼
確認服務恢復 → 撰寫事後分析（PIR）
```

## 7. 軟體更新檢查清單

### 更新前

- [ ] 確認目標版本與現有元件相容性
- [ ] 備份 etcd（`etcdctl snapshot save`）
- [ ] Velero 備份 K8s 資源
- [ ] 確認所有映像 SHA-256 checksum
- [ ] 掃描報告無 CRITICAL 漏洞（或已知例外核准）
- [ ] 更新已在測試叢集驗證通過
- [ ] 維護視窗已通知使用者
- [ ] 回滾計畫已確認

### 更新後

- [ ] 所有 Pod 狀態恢復 Running
- [ ] Run.ai 作業可正常提交
- [ ] GPU 使用率監控正常
- [ ] 告警規則正常運作
- [ ] 更新紀錄填寫完畢
- [ ] 版本追蹤文件更新
