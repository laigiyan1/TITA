# 維運與升級策略

## 1. 概述

本文件說明 NVIDIA Run.ai 平台與 Kubernetes 叢集的日常維運流程、監控告警機制，以及版本升級的標準作業程序（SOP），確保平台穩定性與持續可用性。

---

## 2. 平台維運架構

### 2.1 維運層次

```
┌─────────────────────────────────────────────────────┐
│  L3 平台工程師   │  Run.ai / K8s 核心升級、架構優化   │
├─────────────────────────────────────────────────────┤
│  L2 系統維運     │  節點維護、故障排查、效能調校       │
├─────────────────────────────────────────────────────┤
│  L1 監控值班     │  告警回應、服務重啟、用戶問題轉介   │
└─────────────────────────────────────────────────────┘
```

### 2.2 維運工具清單

| 類別        | 工具              | 用途                                             |
| --------- | --------------- | ---------------------------------------------- |
| 指標採集      | Prometheus      | GPU / K8s 指標蒐集（Run:ai 必要元件，不可移除）               |
| 基礎設施監控    | Grafana         | 節點 CPU / 記憶體 / 磁碟等基礎設施儀表板                      |
| AI 工作負載監控 | Run:ai 原生儀表板    | GPU 使用率、工作負載狀態（v2.25 起取代舊版 Grafana Run:ai 儀表板） |
| 日誌        | Loki + Promtail | 叢集與應用日誌集中                                      |
| 告警        | Alertmanager    | 告警路由、靜音、通知                                     |
| 稽核        | K8s Audit Log   | API Server 操作紀錄                                |
| 配置管理      | Helm + ArgoCD   | 宣告式配置版控與自動同步                                   |
| 備份        | Velero          | K8s 資源與 PVC 快照備份                               |

> ⚠️ **Run:ai v2.25 變更**：Legacy Grafana 儀表板（Overview、Analytics、Consumption、Multi-cluster）已自 v2.25 起移除，**Run:ai GPU 工作負載管理統一改由 Run:ai 原生介面呈現**。Grafana 仍可用於節點層基礎設施監控，但不再是 Run:ai GPU 管理的主要儀表板工具。  
> 來源：[What's New v2.25](https://run-ai-docs.nvidia.com/self-hosted/getting-started/whats-new/whats-new-2-25)

## 3. 監控指標與告警規則

### 3.1 GPU 層指標（DCGM Exporter）

| 指標名稱 | 說明 | 告警閾值 |
|----------|------|----------|
| `DCGM_FI_DEV_GPU_UTIL` | GPU 運算使用率 | < 10% 持續 30 分鐘（可能閒置） |
| `DCGM_FI_DEV_MEM_COPY_UTIL` | GPU 記憶體頻寬使用率 | > 95% 持續 5 分鐘 |
| `DCGM_FI_DEV_GPU_TEMP` | GPU 溫度 | > 85°C |
| `DCGM_FI_DEV_POWER_USAGE` | GPU 功耗 | > TDP × 0.95 |
| `DCGM_FI_DEV_ECC_SBE_VOL_TOTAL` | ECC 單位元錯誤累計 | > 100（警告）|
| `DCGM_FI_DEV_ECC_DBE_VOL_TOTAL` | ECC 雙位元錯誤 | > 0（立即告警）|

### 3.2 Kubernetes 層指標

| 指標 | 告警條件 |
|------|----------|
| Node NotReady | 節點離線 > 5 分鐘 |
| PVC Pending | 儲存空間不足 |
| Pod CrashLoopBackOff | 元件重啟 > 5 次 |
| API Server 延遲 | p99 > 1 秒 |
| etcd 磁碟 fsync | > 100ms |

### 3.3 Run.ai 層指標

| 指標 | 告警條件 |
|------|----------|
| 作業佇列等待時間 | 平均 > 2 小時 |
| GPU 配額超用 | Department 超過核准配額 |
| Scheduler 元件健康 | Run.ai Pod 非 Running 狀態 |

## 4. 日常維運 SOP

### 4.1 每日健康檢查（自動化）

```bash
# 節點狀態
kubectl get nodes -o wide

# Run.ai 元件健康
kubectl get pods -n runai -o wide

# GPU 使用率概覽
runai list jobs --all-projects

# 佇列狀態
runai describe project --all
```

### 4.2 節點進入維護模式

```bash
# 1. 驅逐節點上的工作負載（Run.ai 會自動重排）
kubectl cordon <node-name>
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data

# 2. 執行維護（韌體更新、驅動更新等）

# 3. 恢復節點
kubectl uncordon <node-name>
```

### 4.3 GPU 節點故障處理流程

```
偵測到 GPU 錯誤
      │
      ▼
確認錯誤類型（ECC DBE / Xid Error / 驅動崩潰）
      │
      ├── Xid 79（GPU 硬體錯誤）→ 隔離節點，申請 RMA
      ├── Xid 63（行紀憶體頁面退休）→ 記錄退休數量，監控趨勢
      └── 驅動崩潰 → 嘗試重載驅動 → 若失敗則重啟節點
      │
      ▼
通知相關使用者作業狀態
      │
      ▼
更新維護日誌 & 建立問題單
```

## 5. 升級策略

### 5.1 版本相依矩陣

升級前務必確認官方支援矩陣（以下以 **Run.ai v2.25** 為基準）：

| 元件 | 支援版本（Run.ai v2.25） | 備註 |
|------|--------------------------|------|
| Kubernetes | **1.33 / 1.34 / 1.35** | v1.32 已不支援；每次升級前須重新確認官方矩陣 |
| NVIDIA GPU Operator | **25.10 ~ 26.3** | ⚠️ 版本號存在文件不一致，正式建置前請向 NVIDIA Support 確認（詳見備註） |
| Run.ai | **v2.25** | 本文件基準版本 |
| DCGM | 隨 GPU Operator 一同管理 | 無需單獨指定 |
| OpenShift（選用） | 4.18 ~ 4.21 | K8s 叢集環境不適用 |
| Knative（推論選用） | 1.19 ~ 1.21 | KServe InferenceService / NIM 推論服務自動擴縮時需要 |

> **升級前置作業（必要，勿跳過）**
>
> 1. **查閱目標版本 Support Matrix**：每次升級前須以目標 Run.ai 版本的官方矩陣為依據，確認 K8s、GPU Operator、Knative 的相容版本範圍，不可沿用上一版規則。
> 2. **規劃中繼版本**：若現行 K8s 版本與目標版本跨越多個不支援的中間版本（如 1.30 → 1.33），須逐次升級（1.30 → 1.31 → 1.32 → 1.33）；同理，Run.ai 亦可能需要中繼版本過渡。
> 3. **決定升級順序**：一般情況下為 Kubernetes → GPU Operator → Run.ai，但若 Run.ai 新版本對 K8s 有最低版本要求，應確認現行 Run.ai 版本能否在過渡期間的 K8s 版本上正常運作，再排定順序。
>
> ⚠️ **不可套用固定順序**：升級順序需依每次的 Support Matrix 與中繼版本規劃個別評估，直接按「Kubernetes → GPU Operator → Run.ai」執行而未確認相容性，可能導致叢集進入不支援的混合版本狀態。
>
> **文件版本備註**：Run.ai v2.25 官方文件中 GPU Operator 版本有一處不一致（表格寫 25.10–26.3，部分段落仍有 25.3–25.10 的舊文字）。依 Cluster System Requirements 及 v2.25 What's New 雙重佐證，以 **25.10–26.3** 為準，正式標案建議向 NVIDIA 取得書面確認。

### 5.2 Kubernetes 升級 SOP

**預備作業（D-7）**

- [ ] 備份 etcd：`etcdctl snapshot save`
- [ ] 備份 K8s 資源：`velero backup create pre-upgrade`
- [ ] 確認 Run.ai 支援目標 K8s 版本
- [ ] 在測試環境完成升級演練

**升級流程（滾動升級）**

```
1. 升級 Control Plane（一次一個 Master Node）
   kubeadm upgrade plan
   kubeadm upgrade apply v1.XX.X

2. 升級 Worker Node（分批，每批不超過 25%）
   kubectl cordon <node>
   kubectl drain <node> ...
   kubeadm upgrade node
   apt-get upgrade kubelet kubectl
   kubectl uncordon <node>

3. 驗證叢集健康
   kubectl get nodes
   kubectl get pods -A | grep -v Running
```

### 5.3 Run.ai 升級 SOP

**升級前檢查**

```bash
# 確認目前版本
helm list -n runai-backend
helm list -n runai

# 備份 Run.ai CRD 與 ConfigMap
kubectl get crd | grep run.ai
kubectl get configmap -n runai -o yaml > runai-configmap-backup.yaml
```

**升級步驟**

```bash
# 1. 更新 Helm Chart Repository
# ── v2.25 起以 NVIDIA NGC 為主要來源（JFrog registry 已 deprecated）──
#
# NGC Helm repo（依官方文件）：
#   repo URL  : https://helm.ngc.nvidia.com/nvidia/runai
#   CP chart  : runai/control-plane    （release name 慣例為 runai-backend）
#   Cluster chart: runai/runai-cluster （release name 慣例為 runai）
#
# ⚠️ 請於正式建置前向 NVIDIA 確認最新 chart 命名，以官方安裝文件為準

helm repo add runai https://helm.ngc.nvidia.com/nvidia/runai
helm repo update

# 2. 升級 Control Plane（chart: control-plane，release name: runai-backend）
helm upgrade runai-backend runai/control-plane \
  -n runai-backend \
  --version <target-version> \
  -f values-override.yaml

# 3. 升級 Cluster（Scheduler + Agent）
helm upgrade runai runai/runai-cluster \
  -n runai \
  --version <target-version> \
  -f cluster-values-override.yaml

# 4. 驗證升級
kubectl rollout status deployment -n runai
kubectl rollout status deployment -n runai-backend
```

> ⚠️ **過渡注意**：v2.25 之前仍使用 `runai.jfrog.io` 的既有部署，升級至 v2.25 時須先將 Helm repo 切換至 NGC，再執行升級。請勿在 v2.25 後繼續引用 JFrog 路徑。

**升級後驗證**

```bash
# 確認所有 Run.ai Pod 正常
kubectl get pods -n runai
kubectl get pods -n runai-backend

# 提交測試作業
runai submit verify-job --image nvcr.io/nvidia/cuda:12.3.0-base-ubuntu22.04 \
  --gpu 1 --command -- nvidia-smi
```

### 5.4 NVIDIA GPU Operator 升級

```bash
# 查看目前版本
helm list -n gpu-operator

# 升級（注意 values 變更）
helm upgrade gpu-operator nvidia/gpu-operator \
  -n gpu-operator \
  --version <target-version> \
  --set driver.version=<driver-version> \
  -f gpu-operator-values.yaml
```

## 6. 備份與災難復原

### 6.1 備份矩陣

| 備份項目 | 工具 | 頻率 | 保留期限 |
|----------|------|------|----------|
| etcd 快照 | etcdctl | 每 6 小時 | 7 天 |
| K8s 資源（YAML） | Velero | 每日 | 30 天 |
| PVC 資料（模型、資料集） | Velero + CSI Snapshot | 每週 | 90 天 |
| Run.ai 設定（Projects/Departments） | GitOps（ArgoCD） | 即時 | 版控保留 |

### 6.2 RTO / RPO 目標

| 場景 | RTO（恢復時間） | RPO（資料損失容忍） |
|------|----------------|-------------------|
| 單一節點故障 | < 10 分鐘 | 0（運算無狀態） |
| Control Plane 故障 | < 30 分鐘 | < 6 小時（etcd 備份） |
| 全叢集毀損 | < 4 小時 | < 24 小時 |

## 7. 變更管理

所有升級與配置變更遵循以下流程：

```
提出變更請求（CR）
      │
      ▼
技術審查（影響範圍、回滾計畫）
      │
      ▼
排定維護視窗（離峰時段）
      │
      ▼
通知使用者（提前 48 小時）
      │
      ▼
執行變更 + 即時監控
      │
      ├── 成功 → 關閉 CR，更新文件
      └── 失敗 → 執行回滾 → 事後分析（PIR）
```
