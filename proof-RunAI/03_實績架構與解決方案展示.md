# 實績架構與解決方案展示

## 1. 概述

本文件展示 NVIDIA Run.ai 結合 Kubernetes 的典型部署架構，並以具體場景說明平台如何解決 AI 基礎設施的核心痛點，作為專案可行性評估的參考依據。

## 3. 解決方案場景展示

### 3.1 場景一：多團隊 GPU 資源競搶問題

**痛點**

> 三個研究團隊共用 32 張 GPU，資源分配依賴人工協調，常發生搶佔衝突，GPU 閒置率高達 40%。

**解決方案**

```
Department: AI Research  (GPU Quota: 32)
  ├── Project: NLP          Quota: 12, Used: 10, Queued: 3 jobs
  ├── Project: ComputerVision  Quota: 12, Used: 8,  Queued: 0 jobs
  └── Project: Reinforcement  Quota: 8,  Used: 4,  Queued: 1 job
                                           ↑
                          ComputerVision 閒置 4 GPU 自動借用給 NLP
```

- NLP 團隊配額 12 GPU，但目前 ComputerVision 僅用 8（配額 12），系統自動允許 NLP **Over-Quota 使用至 16 GPU**
- 當 ComputerVision 提交新作業，自動搶回配額，NLP 超用部分進入佇列

**成效**

| 指標 | 導入前 | 導入後 |
|------|--------|--------|
| GPU 平均使用率 | 62% | 83% |
| 等待協調時間 | 平均 2 小時 | 自動排隊 |
| 資源分配糾紛 | 每週 3–5 件 | 趨近於零 |

### 3.2 場景二：GPU 碎片化浪費

**痛點**

> 大量小型實驗作業（如超參數搜尋）各自申請完整 1 張 GPU，但實際僅使用 15–20%，造成嚴重浪費。

**解決方案：GPU Fraction**

```yaml
# v2.25 寫法：標準 Kubernetes Job + Run.ai Dynamic GPU Fraction
# 超參數搜尋：同一節點同時跑 4 個 Trial，每個佔用 1/4 GPU
apiVersion: batch/v1
kind: Job
metadata:
  name: hpo-trial-001
  namespace: research-team
spec:
  template:
    metadata:
      annotations:
        gpu-fraction: "0.25"         # Run.ai GPU Fraction（Pod 層級，無 runai/ prefix）
        # 若要限制記憶體上限（Dynamic GPU Fractions），可改用：
        # gpu-memory: "10G"          # 以 GPU 記憶體量指定（擇一使用）
    spec:
      schedulerName: runai-scheduler
      containers:
        - name: train
          image: myregistry.local/ml-train:latest
          command: ["python", "train.py", "--lr", "0.001", "--batch", "128"]
          resources:
            limits:
              nvidia.com/gpu: "1"    # GPU slot 佔位
      restartPolicy: Never
```

```
A100 GPU（實體 1 張）
┌─────────────────────────────────────┐
│  Trial-001  │  Trial-002  │         │
│   0.25 GPU  │   0.25 GPU  │ 0.5 GPU │
│             │             │ (空閒可  │
│  Trial-003  │  Trial-004  │  動態借) │
│   0.25 GPU  │   0.25 GPU  │         │
└─────────────────────────────────────┘
```

**成效**：同等 GPU 資源下，超參數搜尋吞吐量提升 3–4 倍。

### 3.3 場景三：大模型分散式訓練

**痛點**

> LLM 訓練需要 64 張 GPU，跨越 8 個節點，手動協調節點分配、InfiniBand 路由複雜。

**解決方案：分散式工作負載**

```yaml
# v2.25 寫法：Kubeflow PyTorchJob 多節點分散式訓練
# 舊版 run.ai/v2alpha1 DistributedWorkload CRD 已不建議使用
apiVersion: kubeflow.org/v1
kind: PyTorchJob
metadata:
  name: llm-pretrain-70b
  namespace: nlp-team
spec:
  pytorchReplicaSpecs:
    Master:
      replicas: 1
      template:
        spec:
          schedulerName: runai-scheduler
          containers:
            - name: trainer
              image: nvcr.io/nvidia/nemo:25.02
              command:
                - torchrun
                - --nproc_per_node=8
                - --nnodes=8
                - pretrain_llm.py
                - --model-size=70B
              env:
                - name: NCCL_IB_DISABLE
                  value: "0"          # 啟用 InfiniBand
                - name: NCCL_DEBUG
                  value: "WARN"
              resources:
                limits:
                  nvidia.com/gpu: "8"
    Worker:
      replicas: 7
      template:
        spec:
          schedulerName: runai-scheduler
          containers:
            - name: trainer
              image: nvcr.io/nvidia/nemo:25.02
              resources:
                limits:
                  nvidia.com/gpu: "8"
```

- Run.ai 自動選擇拓撲最佳節點（同一 Rail / InfiniBand Switch）
- 節點故障時自動重排並恢復 Checkpoint

### 3.4 場景四：推論服務自動擴縮

**痛點**

> 推論服務流量白天尖峰，夜間低谷，固定分配 GPU 導致尖峰不足、離峰浪費。

**解決方案：KServe InferenceService + HPA**

```yaml
# v2.25 寫法：KServe InferenceService 搭配 Run.ai Scheduler
# 舊版 run.ai/v2alpha1 InferenceWorkload CRD 已不建議使用
apiVersion: serving.kserve.io/v1beta1
kind: InferenceService
metadata:
  name: bert-inference
  namespace: inference-prod
spec:
  predictor:
    minReplicas: 1
    maxReplicas: 8
    scaleTarget: 100                 # 目標 100 req/s per replica
    scaleMetric: rps
    pytorch:
      storageUri: s3://models/bert
      runtimeVersion: "25.02"
    podSpec:
      schedulerName: runai-scheduler
      # ⚠️ KServe InferenceService 的 GPU fraction annotation 放置位置
      #    需依 Run.ai Resource Interface 對 KServe 的 mapping 方式驗證；
      #    目前建議做法為在 podSpec 下對應的 Pod template 層級加入：
      #      gpu-fraction: "0.5"     （無 runai/ prefix）
      #    正式部署前請確認 Run.ai RI 文件中 KServe 的 annotation 注入方式。
      containers:
        - name: kserve-container
          image: myregistry.local/triton-bert:latest
          resources:
            limits:
              nvidia.com/gpu: "1"    # GPU slot 佔位
          env:
            # Dynamic GPU Fractions 記憶體限制（擇一方式使用）
            - name: RUNAI_GPU_MEMORY_LIMIT
              value: "20Gi"
```

3. Run.ai License 採離線授權模式（Offline License）
4. 監控資料留存於叢集內部，不對外傳送遙測

## 7. 導入成功關鍵因素

1. **Namespace 設計前置**：Project 對應 Namespace，初期規劃影響後期擴充彈性
2. **儲存層整合**：RWX 儲存（NFS/CephFS）確保多節點作業共享資料集
3. **網路頻寬確認**：分散式訓練需要 InfiniBand 或高速 RoCE，事前規劃避免效能瓶頸
4. **RBAC 與 AD 整合**：使用者帳號管理需在初期完成 OIDC/LDAP 整合
5. **Checkpoint 機制**：搭配 Run.ai 搶佔機制，訓練作業須實作 Checkpoint 以防中斷損失
